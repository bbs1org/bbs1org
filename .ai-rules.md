# bbs1org 插件开发 AI 规则

本文件用于约束 AI 为 bbs1org 开发或修改插件，规则按优先级排列且必须逐条遵守。

## 最高优先级

- 插件必须兼容 PHP 8.1、SQLite、MySQL 和 PostgreSQL，禁止引入框架、Composer 包或构建依赖。
- 插件必须位于 `app/plugins/插件ID/plugin.php`，文件开头必须写 `if (!defined('APP_ROOT')) exit;`。
- 插件 ID 只使用小写字母、数字、下划线或短横线，函数、常量、表名、CSS 类和全局变量必须以插件 ID 命名空间化。
- 插件 manifest 必须准确声明 `id`、`name`、`version`、`description` 和 `author`，可选能力再放入 `assets`、`hooks`、`routes`、`admin_tabs`、`install` 或 `uninstall`。
- 优先通过 Hook、路由和后台标签实现功能，除非插件机制无法满足需求，否则不要修改 `index.php` 或核心资源。
- 修改现有插件时必须兼容旧配置和旧数据，新配置项必须有安全默认值且缺失时不能报错。
- 每次行为变化都必须更新插件版本号，并同步修改 manifest 中的描述。

## 数据库与性能

- 数据库结构必须使用 `app_db_*` 函数，普通参数化查询使用 `q()`、`one()` 或 `val()`。
- 字段类型必须来自 `app_db_types()`，ID、外键、计数和时间使用 `uint`，状态位使用普通 `INTEGER`。
- 插件表必须使用 `plugin_插件ID_` 前缀，系统表必须保持 `app_` 前缀。
- Schema 函数只能由插件的 install 函数调用，禁止在普通请求、页面渲染或业务函数中重复执行 Schema。
- 数据表升级使用 `app_db_ensure_columns()`，索引和卸载分别使用对应的 `app_db_create_index()`、`app_db_drop_index()` 和 `app_db_drop_table()`。
- Upsert 键必须有主键或唯一索引，更新使用 `app_db_upsert()`，仅防重复时使用 `app_db_insert_ignore()`。
- 新增记录后必须使用 `app_db_last_insert_id('表名')`，禁止直接调用 PDO 的 `lastInsertId()`。
- 多个时间表达式取最大值必须使用 `app_db_greatest()`，禁止使用仅适配单一数据库的 SQL 写法。
- 多步关联写入必须使用 `tx()` 保证原子性，失败时不得留下半完成数据。
- 禁止在主题、回帖、用户或后台列表循环中逐条查询数据库，必须合并 ID 后批量读取并建立映射。
- 同一请求重复读取的数据必须使用请求级缓存，允许延迟的数据可使用短期 Session 缓存。
- 数据发生新增、更新或删除后必须主动失效相关缓存，禁止为验证缓存额外查询数据库。
- 新增或更新主题后必须调用 `topic_fts_sync()`，新增或更新回帖后必须调用 `reply_fts_sync()`。
- 禁止直接读写 `app_topics_fts` 或 `app_replies_fts`，搜索索引差异必须交给核心兼容层处理。
- 需要保存可搜索的结构化正文时优先使用标准 Markdown 表格，禁止用 Base64 或私有编码隐藏可搜索内容。

## Hook、页面与资源

- Hook 函数通常接收 `($value, array $ctx)` 并返回修改后的值，返回 `null` 表示不修改。
- Hook 必须优先复用 `$ctx` 和 `$value` 中的已有数据，禁止为每个渲染项重复查询相同信息。
- 插件识别自有主题或回帖时应先检查专属内容标识，只有命中后才允许查询插件表。
- 固定 CSS 和 JavaScript 必须通过 manifest 的 `assets` 声明，禁止通过 `page.head` 或 `page.footer` 注入 `<style>` 或内联 `<script>`。
- CSS 资源函数只能返回 CSS 源码，JavaScript 资源函数只能返回 JavaScript 源码，两者都不得输出标签。
- 合并资源不得依赖当前用户、页面、CSRF 或实时请求数据，动态值必须通过插件 HTML 的 `data-*` 属性传递。
- 禁止直接修改自动生成的 `app/assets/plugins.css` 和 `app/assets/plugins.js`，资源变化必须通过插件 assets 和重建资源完成。
- 前台页面必须通过 manifest 的 `routes` 注册并使用 `route_url()` 生成链接，禁止硬编码 `index.php` 查询串。
- 后台页面必须通过 manifest 的 `admin_tabs` 注册，表单操作必须使用 POST、`form_token()` 和核心权限检查。
- 所有输出到 HTML 的外部或用户数据必须使用 `h()` 转义，URL 也必须使用核心 URL 函数生成后再转义。
- 插件页面必须覆盖无数据、加载失败、权限不足和移动端布局，不得只实现理想状态。

## 配置、安全与外部采集

- 配置读取必须集中归一化类型、枚举、长度和上下限，保存前不得直接信任 `$_POST`、`$_GET` 或 JSON。
- 涉及发帖、回帖、管理或用户数据的路由必须显式执行对应的登录、权限和 CSRF 检查。
- 文件路径必须使用 `DATA_DIR`、`CACHE_DIR`、`PLUGIN_DIR` 或 `UPLOAD_DIR`，公开附件地址必须使用 `upload_url()`。
- 上传和远程文件必须限制协议、主机、类型和大小，并拒绝内网地址、凭据 URL、脚本文件和路径穿越。
- 外部 HTTP 请求必须设置连接超时、总超时、重定向上限、响应大小和明确的 User-Agent。
- Cookie、Token、密码和密钥不得写入页面、日志、错误信息、队列键或可公开访问的文件。
- 定时采集必须使用互斥锁、可重试队列和唯一来源键，并保证重复执行不会重复入库。
- 外部数据再次采集时必须与已有记录比较，只有字段变化时才更新内容、搜索索引和 `updated_at`。
- 外部数据记录必须保存来源 ID、来源 URL、首次创建时间、最后看到时间和最后更新时间。
- 采集正文、图片或回帖必须与配置语义一致，关闭采集时不得先请求数据再丢弃。
- 删除或卸载只能处理插件明确拥有的数据，禁止连带删除无法确认归属的用户内容或文件。

## 代码质量与交付

- 优先使用项目已有核心函数和相邻插件的成熟模式，禁止复制一套功能重复的基础设施。
- 代码必须保持原生 PHP 风格、参数类型明确、分支可读且错误信息简短，不要为了压缩行数牺牲可维护性。
- 新功能必须避免增加普通页面查询数、外部请求或同步阻塞，耗时工作应放入队列或显式执行入口。
- 完成修改后必须运行 `php -l app/plugins/插件ID/plugin.php` 和 `git diff --check`。
- 涉及配置、周期、解析或格式化的纯函数必须至少验证默认值、非法输入和边界值。
- 涉及数据库的改动必须检查占位符数量、唯一约束、索引、事务和三种数据库的 SQL 兼容性。
- 交付说明必须列出行为变化、迁移影响、验证结果和未执行的外部副作用操作。
